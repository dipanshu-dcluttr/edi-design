PO to ASN runtime flow, with ERP punch, Sheet sync, queues, and database state.

This site visualizes one complete example, not just component boxes. The example assumes Blinkit sends a purchase order webhook, our seller workflow runs validation, performs both ERP punch and sheet sync, sends PO acknowledgement to Blinkit, starts ASN tracking, polls ERP until ASN details are ready, creates a canonical ASN, validates it, and sends ASN to Blinkit.

Example business document

PO-1001 is the external Blinkit PO number. The aggregate key is blinkit:seller_123:purchase_order:PO-1001. Every queue message, workflow row, action attempt, retry, ASN tracking job, and audit entry carries this readable key plus aggregate_key_hash so same-PO processing is serialized and traceable.

Design intent

The core workflow engine owns state. Plugins do work. Queues carry work. Postgres records truth. Approval, validation, ERP, sheet, ack, ASN scheduling, ASN source polling, and ASN sync are all workflow actions or infrastructure driven by workflow state.

ASN payload creation also freezes onboarding config: seller warehouse, tax profile, item mapping, and ASN source config are captured in an ASN build snapshot before the ASN is sent.

          Database table schema

          The schema is split between immutable evidence, runtime state, action execution, and long-running jobs. The important design rule is that queues never become the source of truth. A queue message only points to these rows; the worker locks the row, verifies state, executes, and writes the next state transition.

          Database rows after the happy path

          These rows show how the same business event becomes traceable state. IDs are shortened for readability, but the relationships are accurate: raw event creates document/version, document creates workflow run, workflow run creates node runs, node runs create action attempts, action attempts create validation results, partner messages, approval tasks, and ASN tracking jobs.

          Queue schema and routing contract

          The queue schema is deliberately thin. Messages include the identity needed to find durable state and execute exactly one unit of work. Large payloads stay in object storage or Postgres references. Retry, DLQ, and replay behavior is driven by action attempt state, not by mutating business data from the queue consumer.

          Queue contract

          Queue messages

          Queue messages are not state. They are commands to workers. The durable state lives in Postgres. Each message carries IDs that let the worker lock the correct row, detect stale attempts, and resume safely after retry.

          User flow and operator visibility

          The UI should not show low-level internals first. It should show a PO journey with drill-down into state, payloads, queue messages, retries, approvals, and ASN tracking jobs. The system keeps internal state detailed enough for engineering while presenting seller operators with clear actions.

          Failure paths

          Failure handling is scoped to the failed unit of work. If sheet sync fails, ERP punch is not repeated. If ASN is not ready, no worker stays alive; `asn_tracking_jobs.next_poll_at` moves forward. If a duplicate PO arrives, no workflow action runs again.

          State model summary

          Raw event state

          • received: webhook stored before parsing.
          • queued: ingestion message created.
          • processed: canonical document accepted.
          • duplicate: exact same idempotency key or payload hash already processed.
          • parse_failed: partner plugin could not map payload.

          Workflow/action state

          • workflow_runs represent one execution of a published workflow version.
          • workflow_node_runs represent each node: validation, ERP, sheet, ack, ASN schedule.
          • action_attempts represent one plugin attempt with retry and idempotency data.
          • Approval is approval.manual_decision, a normal action node.

          ASN tracking job state

          • waiting_for_asn: PO workflow has scheduled ASN tracking.
          • poll_due: poller can enqueue ASN source action.
          • not_ready: ERP says ASN details are not ready; reschedule.
          • ready: canonical ASN can be created.
          • completed: ASN workflow accepted by Blinkit.

          Same PO ordering

          • All events for PO-1001 share one aggregate key.
          • Document worker locks the aggregate before changing document/workflow state.
          • Exact duplicates are marked duplicate.
          • Changed same-PO payload creates a new document version or manual review depending on side effects already completed.
