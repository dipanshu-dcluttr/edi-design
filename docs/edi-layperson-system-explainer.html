EDI Integration Platform, in Plain English

We are building reliable middleware between marketplaces like Blinkit and seller systems like ERP, sheets, and warehouse tools. It receives purchase orders, runs the right business steps, sends confirmations and shipment notices, and keeps proof for every action.

What Are We Building?

Core Story

Imagine Blinkit sends a seller a purchase order. The seller may use ERP, a sheet, or another operations tool. Our system sits in the middle and makes sure the order is received, checked, routed, acknowledged, shipped, and audited.

Receive

Bring every platform message inside safely

Webhook arrives. We authenticate it, store the original payload, assign a trace id, and create a durable record before any business logic runs.

Orchestrate

Run the seller's configured workflow

Validation, ERP punch, sheet sync, approval, PO acknowledgement, ASN tracking, and notifications are workflow nodes, not hidden code paths.

Prove

Show exactly what happened later

Operators can see raw payloads, document versions, action attempts, retries, external requests, responses, and state changes in one journey view.

How We Should Build It

Monolith vs Microservices

Use Modular Monolith + Event-Driven Workers

Start with one backend codebase and one source-of-truth database, but run separate process modes for API, workflow, outbox, ingestion, actions, and ASN polling. This gives clean boundaries without early microservice complexity.

Do this first
Split later only when pain is real

Always Running

  • API ServerReceives Blinkit webhooks and serves ops/config APIs.
  • Outbox DispatcherPublishes queued work after database commits.
  • Ingestion WorkerTurns raw platform payloads into canonical documents.
  • Workflow WorkerAdvances workflow state and decides next actions.
  • ASN SchedulerFinds due ASN tracking jobs and queues poll work.

Event Driven

  • ValidationRuns rules only when workflow reaches validation node.
  • ERP PunchCalls seller ERP when queue has ERP action work.
  • Sheet SyncWrites rows only for sellers configured for sheets.
  • Partner CallsSends Blinkit ack, ASN, and amendment from action jobs.
  • Retry/DLQRetries failed jobs and parks exhausted failures for review.
Why this wins

Fast to build, easy to debug

One repo and one database keep local development and debugging simple while domain rules are still evolving.

What stays central

Truth stays in Postgres

Events trigger work, but document state, workflow state, locks, idempotency, and config snapshots stay DB-owned.

When to split

Split by proven pressure

Move to microservices only when scaling, ownership, isolation, or deployment pain proves a boundary.

same repo / same deployable app api app worker:ingestion app worker:workflow app worker:outbox app worker:action-validation app worker:action-erp app worker:action-partner app worker:asn-poller

System Parts

Architecture

The design keeps partner-specific code, workflow state, side effects, and audit records separate. That gives us flexibility without losing control.

Ingress API

Receives webhooks, checks API key/IP allowlist, detects payload type, stores raw event, and queues ingestion.

Raw Event Writer

Saves original inbound or outbound payload reference before parsing, replay, or business processing.

Idempotency Service

Prevents duplicate processing and duplicate external calls during retries, crashes, and replays.

Partner Plugins

Blinkit-specific parsing and outbound payload building stay outside the workflow core.

Canonical Documents

Common internal PO, ASN, acknowledgement, and amendment shape independent of platform field names.

Workflow Engine

Owns document state and runs published workflow versions with frozen config snapshots.

RabbitMQ + Outbox

Moves work to action workers reliably after database commits, with retries and dead letters.

ASN Poller

Checks ERP or other ASN sources later. Waiting is persisted as job state, not sleeping code.

Why This Design Is Reliable

Guardrails

Guarantees we want

  • Every raw payload is preserved before parsing.
  • Same PO is processed in order.
  • Exact duplicate webhook does not create duplicate side effects.
  • Retries reuse same idempotency key.
  • Running workflows use frozen published config.
  • ASN waiting survives worker restarts and can span days.

Failure paths we handle

  • Blinkit sends duplicate PO.
  • PO changes after side effects already happened.
  • ERP timeout or partial failure.
  • ASN source data not ready yet.
  • Missing item mapping or required ASN fields.
  • Blinkit rejects or partially accepts ASN.

Core Non-Negotiables

What Must Always Be True

These are the rules that make the middleware safe. If any one of these is broken, the system may duplicate orders, lose audit evidence, or become impossible to operate as more brands, ERPs, and platforms are added.

01

DAG workflow, not hardcoded steps

Workflows are graphs. A seller can go PO -> validate -> ERP -> ack, while another seller can go PO -> validate -> sheet -> approval -> ack.

02

Idempotent system end to end

Duplicate webhooks, retries, worker crashes, and replay must not create duplicate ERP orders, duplicate Blinkit acks, or duplicate ASNs.

03

Raw payload stored before parsing

We must always preserve what Blinkit sent and what we sent back, even if parsing or validation fails later.

04

Workflow engine owns state

Action plugins do work and return results. They do not secretly mutate document state. This keeps the system understandable.

05

Plugin-first integrations

Blinkit, future platforms, ERPs, sheets, and notification tools plug in around the core instead of changing the core every time.

06

Long waits are durable jobs

ASN can happen days later. We store waiting state in ASN tracking jobs, not sleeping workers or open workflow actions.

07

Config is versioned and frozen

A workflow run uses the workflow/rules/config version selected at start. New config changes do not rewrite old history.

08

Every action is traceable

Ops must answer: what arrived, what we created, which rule ran, what external call happened, what response came back.

Config Without Chaos

Workflow Rules

Different sellers can run different flows. One seller may punch PO to ERP. Another may only sync a sheet. The workflow graph decides, and the selected published version is frozen for each run.

Inheritance

Global -> Platform -> Vendor -> Seller

More specific config wins. Seller-level workflow overrides vendor/platform/global defaults when available.

Immutable versions

Published workflow cannot change

Draft edits create a new version. Old running workflows keep old behavior and old config snapshot.

Publish validation

Invalid graphs are blocked

System rejects impossible flows, unsafe side effects, missing terminal paths, and ASN sync before ASN exists.

Validation Rulesets

Quality Gates

A validation ruleset is the checklist the workflow uses before it allows a PO or ASN to move forward. For a layperson: it is like a configurable quality gate that says what is allowed, what is suspicious, and what must stop.

Ruleset inheritance

Rules are layered from broad company safety rules down to seller-specific operating rules. Lower layers can add or customize only when the upper layer permits it.

One final checklist per seller, built from inherited layers
GlobalNon-negotiable platform safety: PO number, line items, duplicate line checks.
PlatformBlinkit contract rules: facility, supplier identity, date format, required partner fields.
VendorCustomer business rules: SKU catalog, tax mapping, brand-specific checks.
SellerWarehouse or seller rules: delivery window, minimum order value, allowed overrides.

Block

Hard stop. Example: missing seller mapping or missing PO lines. Workflow routes to rejection/manual review.

Warn

Continue, but record finding. Example: low order value or unusual delivery window.

Observe

Dry-run/shadow mode. Store findings for learning without affecting live flow.

Admin level

Controls safety and contracts

System/platform admins create reusable rule definitions, lock non-negotiable rules, publish ruleset versions, and decide what lower levels may override.

User level

Customizes allowed business rules

Vendor or seller admins can add allowed field/reference/expression rules, adjust severity where permitted, and preview impact before publishing.

1. Pick from rule catalogReusable rule definitions like required field, SKU exists, tax mapping, or delivery date window.
2. Build ruleset draftChoose which rules apply for PO or ASN and set mode, severity, blocking, and config.
3. Preview effective rulesetShow final merged rules after global, platform, vendor, and seller inheritance.
4. Dry-run samplesRun against sample raw/canonical documents before affecting live workflows.
5. Publish versionOnly immutable published versions run. Future workflow runs freeze selected ruleset in config snapshot.

Features We Support

Scope and Future Ease

Because we are middleware, the architecture must handle repeated variation. Every new brand may bring a different ERP, different seller rules, different item mappings, and eventually different platforms beyond Blinkit.

Multi-platform onboarding

Blinkit is first. Future platforms get their own partner plugin for auth, parsing, mapping, and outbound API payloads.

Multi-brand and multi-seller setup

Vendor, seller, warehouse, tax, contact, platform account, and SKU mappings are stored separately so each customer can differ.

Configurable DAG workflows

Drag/drop style workflows can route by seller need: ERP, sheet, approval, notification, ack, ASN scheduling, or amendment.

ERP integration flexibility

ERP punch and ERP ASN readiness are plugins. New brands can bring new ERP APIs without touching platform core.

Validation and approval rules

Rulesets and approval policies can be inherited globally, per platform, per vendor, or per seller.

Operational recovery

Retries, dead letters, manual review, replay, audit logs, and journey views help ops recover safely when integrations fail.

Brand 1Use Blinkit plugin, ERP plugin, default PO and ASN workflows.
Brand 2Reuse core, add different ERP plugin and seller-specific workflow.
Brand 3Reuse core, add sheet route, manual approval, different ASN source policy.
Future platformAdd new platform plugin while preserving workflow, idempotency, audit, and ops model.
