EDI Integration Platform

Multi-platform, multi-vendor, multi-seller document workflow platform. Blinkit/partnersbiz is first platform. Core owns state, idempotency, validation, audit, retries, and traceability. Plugins own partner/API/ERP specifics.

5. Plugin Model

Partner Plugin

  • Parse partner payload JSON/XML
  • Parse enough contract shape to map payload
  • Map to canonical documents
  • Build partner outbound payloads
  • Map partner errors/warnings

Action Plugin

  • Implement one workflow node
  • Can be validation, ERP, sheet, ack, ASN, amendment, notification
  • Prepare request from canonical doc/config
  • Execute external side effect when applicable
  • Interpret response into normalized result
  • Return errors/warnings/status
  • Never mutate document state directly
ActionPlugin:
  capabilities()
  prepare(context)
  execute(request)
  interpret(response)
  compensate(context) optional

6. Workflow Validation, Inheritance, Approval

flowchart LR
  Canonical["Canonical Document"] --> Workflow["Workflow Run"]
  Workflow --> Node["Node: validate_po"]
  Node --> Plugin["Action Plugin\nvalidation.ruleset_engine"]
  Plugin --> Ruleset["Effective Ruleset\nGlobal -> Platform -> Vendor -> Seller"]
  Ruleset --> Findings["Validation Results"]
  Findings --> OK["node success"]
  Findings --> Fail["node rejected"]
  OK --> Next["Next configured action"]
  Fail --> Review["Manual review / fail edge"]

Ruleset

Global rules inherited and overridden by platform, vendor, and seller config.

Approval

Every action has a workflow-owned approval gate before plugin execution.

Edge Policy

Drag/drop workflows are validated before publish; unsafe edges are rejected.

7. Scale, Duplicates, Same PO

flowchart LR
  Many["Many platform events"] --> API["Stateless API replicas"]
  API --> Raw["Raw Event Inbox"]
  Raw --> Idem["Idempotency unique key"]
  Idem --> Q["RabbitMQ ingestion"]
  Q --> Worker["Document workers"]
  Worker --> Lock["Postgres aggregate lock/version"]
  Lock --> State["Single state update per PO"]
  State --> Outbox["Transactional outbox"]
  Outbox --> Actions["Action queues/workers"]
CaseBehavior
Exact duplicateStore raw event, mark duplicate, no new actions.
Same PO changed before required side effectCreate new version and start/restart workflow from configured safe point.
Same PO changed after required side effect/workflow completionDo not overwrite. Manual review or user-created amendment.
Concurrent same PO eventsSame aggregate key, Postgres lock/version serializes updates.
Slow external APIAction queue/retry handles it; ingestion continues.

8. ASN Poller

flowchart LR
  POClosed["Workflow completed"] --> Job["Create asn_job if configured"]
  Job --> Due["next_poll_at due"]
  Due --> ERP["ASN source plugin\nERP by default"]
  ERP --> NotReady["not_ready"]
  NotReady --> Reschedule["Backoff + next_poll_at"]
  ERP --> Incomplete["data_incomplete"]
  Incomplete --> Review["Ops review / reschedule"]
  ERP --> Ready["ASN data ready"]
  Ready --> Canonical["Canonical ASN"]
  Canonical --> Workflow["ASN workflow run"]
  Workflow --> Validate["Validation action"]
  Validate --> Send["ASN sync action\npartnersbiz"]
  Send --> Terminal["accepted / partial / rejected"]

ASN can happen days after the PO webhook is processed. Polling state lives in Postgres, not memory.
