Architecture Summary

Stable Core

Workflow, state, audit, retry, idempotency, validation, event history.

Partner Plugins

Blinkit plugin owns PDF/API contract parsing and outbound payload building.

Action Plugins

Everything after canonical document creation is workflow-owned. Validation, ERP, sheet sync, PO ack, ASN creation, ASN sync, amendment, notification are workflow action nodes.

Scale Model

Many POs in parallel, same PO serialized by aggregate key and Postgres lock.

Use The Example Diagrams

Happy Example shows one Blinkit PO flowing through all four state layers: raw event, PO document, workflow run, and action attempts. Messy Example shows duplicate events, changed PO payloads, validation failures, retryable plugin failures, DLQ/manual review, and ASN errors.

State Lifecycles

State is split by ownership. PO document state stays generic; validation, approvals, ERP, sheet sync, ack, ASN, amendment, and notification live in workflow node/action state.

Scale + Duplicate Strategy

ProblemDesign
High event volumeStateless API and KEDA-scaled workers.
Same PO duplicateUnique idempotency key, mark duplicate, no new actions.
Same PO changedBefore any required side effect: new version. After required side effect/workflow completion: manual review/amendment.
Concurrent same POAggregate key + Postgres lock/optimistic version.
Slow ERP/APIAction queues, retry scheduler, DLQ, circuit breaker.
